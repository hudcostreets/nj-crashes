# Cells API: extend pyramid past r11 (r12, r13, r14)

## Motivation

The cells-api worker pyramid currently covers r6–r11 (base_res=14, raw at r14). The picker (`pickHexResolutionForPixels`) selects res by on-screen pixel target — at zoom levels typical of intra-muni viewing (z≈13.5+) it wants r12+, but client `MAX_PYRAMID_RES=11` clamps it. Result: zooming past muni level stops refining the hex grid.

User feedback (paraphrased): *"we should have proper multi-scale viewer here, including arbitrary zoom into any region."*

The raw r14 fallback path in the worker exists in principle but is unreliable in practice — over a county-sized polygon it loads enough rows to OOM a 128 MB Worker (observed earlier in `useCellsApi`/cells-api session work).

## Approach

Pre-bake `r12` and `r13` into the pyramid alongside r6–r11. r14 is already the raw layer; once the worker can use it as a "pyramid" lookup (no groupby needed since each row is already at r14), we cover r6–r14.

This is a straightforward extension of the existing pipeline (`njdot/cli/cells.py`, see `_build_pyramid_level()` lines 143–230). No schema changes — same per-cell aggregation columns, same top-K monoid, same r4-sharded layout.

## Storage estimate

Existing r14 raw layer: ~70 MB across 31 r4 shards (3.98M geocoded crashes).

| Level | Cells (rough) | File size |
|---|---|---|
| r12 | r14 / 49 ≈ 80k | ~1.5 MB |
| r13 | r14 / 7 ≈ 570k | ~10 MB |
| r14 | (raw) | (existing 70 MB) |

Total addition: ~12 MB. Negligible.

## Changes required

### 1. Pipeline (`njdot/cli/cells.py`)

```diff
-PYRAMID_LEVELS_DEFAULT = (6, 7, 8, 9, 10, 11)
+PYRAMID_LEVELS_DEFAULT = (6, 7, 8, 9, 10, 11, 12, 13)
```

r14 is the raw layer — don't add to `PYRAMID_LEVELS_DEFAULT`. The worker can query the raw layer directly when `res === base_res`; r14-from-raw is a filter-only path (no groupby aggregation), same as a pyramid lookup. See "worker fast-path" below.

### 2. Manifest

`data/cells/manifest.json` is regenerated by `njdot compute cells manifest`. After the new pipeline run, it'll show:
```json
{
  "schema_version": 3,
  "data_version": "{ISO}-{git_sha}",
  "pyramid_levels": [6, 7, 8, 9, 10, 11, 12, 13],
  ...
}
```

`schema_version` does not need to bump — the field shape is unchanged. `data_version` rolls automatically (timestamp + git sha).

### 3. Worker (`cells-api/src/cells.ts`)

Two tweaks:

- **r14 fast-path**: when `res === manifest.base_res`, query the raw layer with no groupby. The current raw-fallback path tries to aggregate raw rows up to the requested res, which is wasted work (and the OOM risk) when target equals source. Add a branch that just filters by `(year, severity)` and returns rows directly.
- **Pyramid level dispatch**: no change needed — `manifest.pyramid_levels.includes(res)` already routes correctly. Once the manifest lists 12 and 13, the worker will use the pyramid files for them automatically.

### 4. Client

```diff
- const MAX_PYRAMID_RES = 11
+ const MAX_PYRAMID_RES = 14
```

in `www/src/map/useCellsApi.ts`. The client picker will then walk to r14 when zoom warrants it.

`H3_RADIUS_METERS` in `www/src/map/StackedHexLayer.tsx` already covers r12 (9.4 m), r13 (3.6 m); add r14 (1.4 m) — one line.

### 5. Cells cap interaction

`CELLS_CAP=100000` is already large enough for r12/r13 over any single muni viewport. Statewide r12 would blow it (NJ ≈ 22000 km², r12 ≈ 71M cells), but that scenario can't occur in practice — picker won't pick r12 at statewide-level zoom. No cap change needed.

## Build / push (EC2)

Run from the repo root (with crashes pipeline env active):

```bash
# 1. Bump PYRAMID_LEVELS_DEFAULT in njdot/cli/cells.py
# 2. Re-run pyramid build (uses existing r14 raw layer, only writes new r12/r13 levels)
njdot compute cells pyramid -f

# 3. Regenerate manifest
njdot compute cells manifest

# 4. Push to R2 (aws s3 sync; only new files transferred)
njdot compute cells push

# 5. Update DVX
dvx commit data/cells/pyramid.dvc
dvx push
```

Expected runtime: ~5–10 min (mostly r12, r13 aggregation; r14 already exists).

After push, the worker picks up the new manifest on its next request (manifest is cached for the page lifetime, but `data_version` rolls so client ETags invalidate automatically).

## Out of scope

- Statewide r12+ rendering. Not a real use case; picker auto-bounds.
- Hover/click drill-down to specific crashes at finer res. The existing `topK` field flows through unchanged; UI surfacing it is separate work.
